# 웹/앱 기획서 섹션별 작성 원칙

## 1. 프로젝트 개요 (Executive Summary)

### 작성 목적
- 의사결정권자가 1분 안에 프로젝트를 파악하도록
- 핵심 가치와 차별점을 명확히 전달

### 필수 포함 요소
- **프로젝트명**: 기억하기 쉽고 서비스 성격을 반영
- **한줄 요약**: 20자 이내, "누구를 위한 무엇" 형식
- **프로젝트 유형**: 신규 개발 / 리뉴얼 / 기능 추가
- **예상 기간**: 전체 개발 기간

### 작성 예시
```
**프로젝트명**: Lunch Pick (오늘의 점심)
**한줄 요약**: 직장인을 위한 AI 점심 메뉴 추천 서비스
**유형**: 신규 개발 (MVP)
**예상 기간**: 3개월 (2025.01 ~ 2025.03)
```

### 흔한 실수
- 너무 길고 장황한 설명
- 프로젝트명이 서비스와 무관
- 한줄 요약에 기능 나열

## 2. 문제 정의 및 해결책 (Problem & Solution)

### 작성 목적
- "왜 이 서비스가 필요한가?"에 대한 논리적 답변
- 문제 → 해결책 → 효과의 인과관계 설명

### 필수 포함 요소
- **현황 분석**: 데이터 기반 시장/사용자 상황
- **Pain Point**: 구체적인 문제점 (정량화)
- **해결 방향**: 우리 서비스의 핵심 가치
- **Why Now**: 시장 타이밍의 적절성

### 작성 예시
```
### 문제 정의
**현황**: 직장인 78%가 매일 점심 메뉴 결정에 고민 (출처: 2024 직장인 식사 실태조사)
**Pain Point**:
- 메뉴 결정에 평균 15분 소요
- 팀원 간 의견 충돌로 스트레스 발생
- 매번 같은 메뉴만 먹는 고착화

### 해결책
**핵심 가치**: 개인 취향과 그룹 조화를 고려한 스마트 추천
- 개인 선호도 학습 기반 맞춤 추천
- 그룹 투표 기능으로 의견 조율
- 히스토리 분석으로 메뉴 다양화

### Why Now
- 코로나 이후 오피스 복귀로 단체 식사 증가
- AI 추천 기술 성숙도 향상
- MZ세대의 "결정 피로" 해소 니즈 급증
```

### 흔한 실수
- 문제를 데이터 없이 추상적으로 서술
- 해결책이 문제와 연결되지 않음
- "지금 해야 하는 이유" 불명확

## 3. 타겟 사용자 (Target Users)

### 작성 목적
- 명확한 사용자 정의로 기능 우선순위 결정
- 마케팅/커뮤니케이션 전략 수립 기반

### 필수 포함 요소
- **주요 타겟 (Primary)**: 구체적 페르소나
- **보조 타겟 (Secondary)**: 확장 가능한 사용자층
- **사용자 니즈**: 타겟별 핵심 니즈
- **사용 시나리오**: 구체적인 사용 상황

### 페르소나 작성 예시
```
### Primary Target: 김대리 (32세, IT기업 마케터)
- **인구통계**: 서울 강남 소재 IT기업, 연봉 5천만원
- **상황**: 10명 규모 팀, 주 3회 팀 점심
- **Pain Point**: 매번 "뭐 먹지?" 질문이 피로함
- **기대 가치**: 빠른 결정, 팀원 모두가 만족하는 선택
- **사용 빈도**: 주 5회 (매일 점심)

### Secondary Target: 박사장 (45세, 50인 기업 대표)
- **상황**: 회식/접대 장소 선정 어려움
- **Pain Point**: 상황에 맞는 식당 찾기 번거로움
- **기대 가치**: VIP 접대에 적합한 고급 식당 추천
```

### 시장 규모 추정 (TAM/SAM/SOM)
```
- **TAM (전체 시장)**: 국내 직장인 2,000만명
- **SAM (접근 가능)**: 수도권 화이트칼라 500만명
- **SOM (초기 목표)**: 강남 IT기업 직장인 50만명
  → 1년차 목표: 5만명 (SOM 10%)
```

### 흔한 실수
- "모든 직장인"처럼 너무 넓은 타겟
- 페르소나 없이 인구통계만 나열
- 시장 규모 추정 없음

## 4. 핵심 기능 (Key Features)

### 작성 목적
- 개발팀이 바로 착수할 수 있는 명세 수준
- 우선순위에 따른 리소스 배분 가이드

### MoSCoW 기법 적용
```
### Must Have (MVP 필수)
1. **회원가입/로그인**: 소셜 로그인 (카카오, 구글)
2. **선호도 설정**: 음식 종류, 맵기, 알러지 등록
3. **오늘의 추천**: 개인화된 3개 메뉴 추천
4. **식당 정보**: 추천 메뉴를 파는 근처 식당 표시

### Should Have (가능하면 포함)
5. **그룹 투표**: 팀원들과 함께 메뉴 선정
6. **히스토리**: 최근 먹은 메뉴 기록
7. **알림**: 점심시간 전 추천 알림

### Could Have (향후 확장)
8. **예약 연동**: 식당 예약 기능
9. **리뷰 시스템**: 사용자 평점/리뷰
10. **기업용 대시보드**: 식대 관리 기능
```

### User Story 형식
```
- 사용자는 [빠른 가입을 위해] [소셜 로그인으로 회원가입]할 수 있다.
- 사용자는 [맞춤 추천을 받기 위해] [음식 선호도를 설정]할 수 있다.
- 사용자는 [오늘 점심을 결정하기 위해] [AI 추천 메뉴 3개를 확인]할 수 있다.
- 사용자는 [팀 점심을 정하기 위해] [그룹 투표를 생성]할 수 있다.
```

### 기능별 상세 명세 예시
```
### F001: 오늘의 추천
- **설명**: 사용자 선호도 기반 점심 메뉴 3개 추천
- **입력**: 사용자 선호도, 최근 식사 히스토리, 현재 위치
- **출력**: 메뉴명, 추천 이유, 근처 식당 리스트
- **규칙**:
  - 최근 3일 내 먹은 메뉴는 제외
  - 날씨 고려 (추우면 국물류 가중치 상승)
- **UI**: 카드 스와이프 형태
```

### 흔한 실수
- 모든 기능이 Must Have로 분류됨
- User Story 없이 기능명만 나열
- 기능 간 우선순위 불명확

## 5. 비즈니스 모델 (Business Model)

### 작성 목적
- 수익 창출 방안 명확화
- 투자자/의사결정권자 설득

### 필수 포함 요소
- **수익 모델**: 어떻게 돈을 버는가
- **가격 정책**: 플랜별 가격
- **수익 예측**: 월/연 예상 매출
- **BM 다각화**: 2개 이상 수익원

### 수익 모델 유형
```
1. **프리미엄 구독 (B2C)**
   - 무료: 기본 추천 3회/일
   - 프리미엄: 9,900원/월 (무제한 추천, 그룹 기능)

2. **기업 구독 (B2B)**
   - 팀 플랜: 5만원/월 (20인 이하)
   - 기업 플랜: 협의 (20인 초과, 식대 관리 포함)

3. **제휴 수수료**
   - 식당 예약 시 수수료 5%
   - 배달 연동 시 수수료 3%
```

### 수익 예측 예시
```
### 1년차 수익 예측
| 항목 | 산정 기준 | 월 예상 |
|------|----------|---------|
| 프리미엄 B2C | 5,000명 × 9,900원 × 전환율 5% | 247만원 |
| 기업 B2B | 50개사 × 5만원 | 250만원 |
| 제휴 수수료 | 일 1,000건 × 1만원 × 5% | 150만원 |
| **월 합계** | | **647만원** |
| **연 합계** | | **7,764만원** |
```

### 흔한 실수
- "광고" 단일 BM
- 수익 예측 없이 모델만 나열
- 비현실적인 전환율 가정 (예: 50%)

## 6. 기술 스택 및 MVP (Tech Stack & MVP)

### 작성 목적
- 개발팀과 기술적 합의
- MVP 범위 명확화

### 필수 포함 요소
- **프론트엔드**: 웹/앱 기술
- **백엔드**: 서버/DB
- **인프라**: 호스팅/배포
- **MVP 범위**: 첫 출시 기능

### 기술 스택 예시
```
### Frontend
- **웹**: React + TypeScript
- **앱**: React Native (iOS/Android 동시)

### Backend
- **서버**: Node.js + Express
- **DB**: PostgreSQL (메인) + Redis (캐시)
- **인증**: Firebase Auth

### Infrastructure
- **호스팅**: AWS (EC2, RDS)
- **CDN**: CloudFront
- **CI/CD**: GitHub Actions

### AI/ML
- **추천 엔진**: OpenAI API (초기) → 자체 모델 (향후)
```

### MVP 범위 정의
```
### MVP (3개월, Phase 1)
- 소셜 로그인
- 선호도 설정
- AI 메뉴 추천 (3개)
- 식당 지도 표시

### Phase 2 (6개월)
- 그룹 투표
- 히스토리/통계
- 알림 기능

### Phase 3 (12개월)
- 예약 연동
- 기업용 대시보드
- 자체 추천 모델
```

### 흔한 실수
- MVP에 모든 기능 포함 (과잉)
- AI가 핵심인데 구현 계획 없음
- 기술 선택 이유 불명확

- 리소스 배분 및 일정 관리

### 마일스톤 예시
```
### Phase 1: MVP 개발 (12주)
```

## 9. 기대 효과 및 성공 지표 (KPI)

### 작성 목적
- 프로젝트 성공 기준 정의
- 측정 가능한 목표 설정

### SMART 목표 설정
```
### 정량 지표 (Primary KPI)
| 지표 | 1개월 | 3개월 | 6개월 | 12개월 |
|------|-------|-------|-------|--------|
| MAU | 1,000 | 10,000 | 50,000 | 100,000 |
| DAU | 300 | 3,000 | 15,000 | 30,000 |
| 프리미엄 전환율 | 3% | 5% | 7% | 10% |
| 월 매출 | 30만 | 250만 | 700만 | 2,000만 |

### 정성 지표 (Secondary KPI)
- 앱스토어 평점: 4.5 이상 유지
- 사용자 만족도 (NPS): 50 이상
- 미디어 노출: 월 3건 이상

### 측정 방법
- Google Analytics, Firebase Analytics
- 앱 내 만족도 조사 (월 1회)
- NPS 조사 (분기 1회)
```

### 흔한 실수
- 측정 불가능한 목표 ("최고의 서비스")
- 숫자 없이 정성적 표현만
- 목표 시점 불명확
