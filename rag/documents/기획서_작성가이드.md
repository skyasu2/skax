# 웹/앱 서비스 기획서 작성 가이드

## 1. 기획서의 본질

기획서는 단순한 아이디어의 나열이 아닌, **"왜 이 프로젝트를 해야 하는가?"**에 대한 논리적인 설득 과정입니다.
읽는 사람(의사결정권자, 개발자, 디자이너)이 명확하게 이해하고 행동할 수 있도록 작성해야 합니다.

### 1.1 좋은 기획서의 특징
- **논리적 흐름**: 문제 → 해결책 → 효과가 자연스럽게 연결됨
- **구체적 수치**: "많은 사용자" 대신 "월 10만 MAU"
- **실행 가능성**: 바로 개발에 착수할 수 있는 수준의 명세
- **리스크 인식**: 예상 문제와 대응 방안 포함

### 1.2 나쁜 기획서의 특징
- 추상적인 표현만 나열 ("혁신적인", "편리한", "최고의")
- 숫자나 근거 없는 주장
- 기술적 실현 가능성 미검토
- 비즈니스 모델 불명확

---

## 2. 필수 포함 요소 (9개 섹션)

### 2.1 프로젝트 개요
- **프로젝트명**: 기억하기 쉽고 의미 있는 이름
- **한줄 요약**: 20자 이내로 핵심 가치 전달
- **프로젝트 유형**: 신규 개발 / 리뉴얼 / 기능 추가
- **예상 기간**: 전체 개발 기간 (예: 3개월)

### 2.2 문제 정의 및 해결책
- **현황 분석**: 현재 시장/사용자의 상황
- **Pain Point**: 구체적인 문제점 (데이터 기반)
- **해결 방향**: 우리 서비스가 제공하는 가치
- **Why Now**: 왜 지금 이 서비스가 필요한가

### 2.3 타겟 사용자
- **주요 타겟**: 구체적인 페르소나 (나이, 직업, 상황)
- **사용자 니즈**: 타겟이 원하는 것
- **사용 시나리오**: 언제, 어디서, 어떻게 사용하는가
- **시장 규모**: TAM/SAM/SOM 추정

### 2.4 핵심 기능
- **필수 기능 (Must Have)**: MVP에 반드시 포함
- **권장 기능 (Should Have)**: 가능하면 포함
- **선택 기능 (Could Have)**: 추후 확장 고려
- **User Story 형식**: "사용자는 [목적]을 위해 [기능]을 할 수 있다"

### 2.5 비즈니스 모델
- **수익 모델**: 구체적인 과금 방식
- **가격 정책**: 플랜별 가격 (무료/프리미엄/기업용)
- **수익 예측**: 월/연 예상 매출
- **BM 다각화**: 최소 2개 이상의 수익원

### 2.6 기술 스택 및 MVP
- **프론트엔드**: 웹/앱 기술 선택
- **백엔드**: 서버/DB 기술 선택
- **MVP 범위**: 3개월 내 출시 가능한 핵심 기능
- **기술적 제약**: 알려진 기술적 한계

### 2.7 일정 및 마일스톤
- **Phase 1**: 기획/디자인 (기간)
- **Phase 2**: 개발 (기간)
- **Phase 3**: 테스트/QA (기간)
- **Phase 4**: 출시/운영 (기간)

### 2.8 리스크 및 대응 방안
- **기술 리스크**: 구현 난이도가 높은 기능
- **비즈니스 리스크**: 시장 반응 불확실성
- **운영 리스크**: 서버 비용, 인력 등
- **대응 전략**: 각 리스크별 완화 방안

### 2.9 기대 효과 및 성공 지표
- **정량 지표**: MAU, DAU, 전환율, 매출
- **정성 지표**: 사용자 만족도, 브랜드 인지도
- **측정 방법**: 어떻게 측정할 것인가
- **목표 시점**: 언제까지 달성할 것인가

## 3. 작성 톤앤매너

### 3.1 명확성
- 모호한 표현("~할 수도 있다") 금지
- 단정적인 표현("~한다", "~이다") 사용
- 전문 용어는 설명과 함께 사용

### 3.2 간결성
- 불필요한 미사여구 제거
- 한 문장에 하나의 정보만
- 핵심만 전달

### 3.3 구조화
- 줄글보다는 개조식(Bullet point) 활용
- 넘버링으로 순서와 우선순위 표현
- 표와 차트로 복잡한 정보 시각화

## 4. 흔한 실수와 해결 방법

### 4.1 "광고" 단일 BM
**문제**: 수익 모델이 "광고"뿐
**해결**: B2B 구독, 프리미엄 플랜, 수수료 모델 등 다각화

### 4.2 MVP 과잉
**문제**: 첫 버전에 모든 기능 포함
**해결**: 핵심 3개 기능만, 나머지는 Phase 2

### 4.3 타겟 불명확
**문제**: "모든 사람"이 타겟
**해결**: 구체적 페르소나 설정 (예: "30대 직장인 여성, 맞벌이")

### 4.4 숫자 부재
**문제**: "많은", "빠른", "효율적인" 등 형용사만
**해결**: 구체적 수치 (MAU 10만, 응답시간 1초 이내)

### 4.5 경쟁사 분석 부재
**문제**: 차별점 불명확
**해결**: 경쟁사 3개 이상 분석, 우리만의 차별점 명시

## 5. 기획서 품질 체크포인트

### 5.1 논리성 (40%)
- [ ] "왜 이걸 해야 하는가?"에 대한 답이 명확한가?
- [ ] 문제 → 해결책 → 효과가 논리적으로 연결되는가?
- [ ] 주장에 근거(데이터, 사례)가 있는가?

### 5.2 실현 가능성 (30%)
- [ ] 기술적으로 구현 가능한가?
- [ ] MVP 범위가 3개월 내 출시 가능한가?
- [ ] 비즈니스 모델이 최소 2개 이상인가?

### 5.3 구체성 (20%)
- [ ] 정량적 목표(MAU, 매출)가 있는가?
- [ ] 타겟 사용자가 구체적인가?
- [ ] 기능 명세가 개발 착수 가능한 수준인가?

### 5.4 완성도 (10%)
- [ ] 9개 필수 섹션이 모두 있는가?
- [ ] 리스크 분석이 있는가?
- [ ] 마일스톤이 구체적인가?
